# ⚡ Historical Energy Module – Complete Feature Documentation

---

## 1. Purpose of This Module

The Historical Energy Module converts **1-second live meter data** into:

- ✅ Shift-wise Energy
- ✅ Hourly Energy
- ✅ Daily Energy
- ✅ Monthly Energy
- ✅ Custom Date Range Energy
- ✅ Machine-wise Consumption
- ✅ Energy Reports & Exports

This module is **required for audits, power analysis, production cost calculation, and optimization.**

---

## 2. Data Source – energy_readings Table

### Table: `energy_readings`

| Column | Description |
|--------|-------------|
| id | Primary Key |
| block_name | Machine Name (M21, M22, M23, C2) |
| total_active_energy | Cumulative kWh |
| total_reactive_energy | Cumulative kVArh |
| total_apparent_energy | Cumulative kVAh |
| voltage_l1_l2 | Voltage |
| effective_power | Power (kW) |
| timestamp | Stored every 1 second |

---

## 3. Core Rule of Energy Calculation (MOST IMPORTANT)

This system stores **CUMULATIVE ENERGY VALUES**.

> ✅ Energy Consumption is ALWAYS:

```
Consumption = End Reading – Start Reading
```

❌ NEVER use SUM  
❌ NEVER use AVG for energy  
✅ ALWAYS use DELTA

## 4. Historical Energy by Custom Date & Time

### 4.1 Start Reading Query

```sql
SELECT *
FROM energy_readings
WHERE block_name = 'M24'
  AND timestamp >= '2025-12-10 10:00:00'
ORDER BY timestamp ASC
LIMIT 1;
```

### 4.2 End Reading Query

```sql
SELECT *
FROM energy_readings
WHERE block_name = 'M24'
  AND timestamp <= '2025-12-10 18:00:00'
ORDER BY timestamp DESC
LIMIT 1;
```

### 4.3 Final Delta Calculation

```sql
SELECT 
  e2.block_name,
  e1.timestamp AS start_time,
  e2.timestamp AS end_time,
  (e2.total_active_energy - e1.total_active_energy) AS active_energy,
  (e2.total_reactive_energy - e1.total_reactive_energy) AS reactive_energy,
  (e2.total_apparent_energy - e1.total_apparent_energy) AS apparent_energy
FROM 
  (SELECT * FROM energy_readings WHERE block_name='M24' AND timestamp >= '2025-12-10 10:00:00' ORDER BY timestamp ASC LIMIT 1) e1,
  (SELECT * FROM energy_readings WHERE block_name='M24' AND timestamp <= '2025-12-10 18:00:00' ORDER BY timestamp DESC LIMIT 1) e2;
```

## 6. Hourly Energy Consumption

```sql
SELECT 
  date_trunc('hour', timestamp) AS hour,
  MAX(total_active_energy) - MIN(total_active_energy) AS energy_used
FROM energy_readings
WHERE block_name = 'M24'
AND timestamp BETWEEN '2025-12-10 00:00:00' AND '2025-12-10 23:59:59'
GROUP BY hour
ORDER BY hour;
```


## 17. Background Scheduler Logic

Runs every night at 12:05 AM:

1. Read yesterday's max-min
2. Insert into daily summary
3. Clear old raw data if required

### Python Scheduler Example:

```python
import schedule
import time
from datetime import datetime, timedelta

def calculate_daily_energy():
    yesterday = datetime.now() - timedelta(days=1)
    start_date = yesterday.replace(hour=0, minute=0, second=0)
    end_date = yesterday.replace(hour=23, minute=59, second=59)
    
    machines = ['M21', 'M22', 'M23', 'M24', 'C2']
    
    for machine in machines:
        # Get start and end readings
        start_reading = get_reading(machine, start_date, 'ASC')
        end_reading = get_reading(machine, end_date, 'DESC')
        
        # Calculate delta
        energy_used = end_reading['total_active_energy'] - start_reading['total_active_energy']
        
        # Insert into daily summary
        insert_daily_summary(machine, yesterday.date(), energy_used)
        
    print(f"✅ Daily energy calculated for {yesterday.date()}")

# Schedule task
schedule.every().day.at("00:05").do(calculate_daily_energy)

while True:
    schedule.run_pending()
    time.sleep(60)
```

## 18. Data Validation Rules

✅ End reading must exist  
✅ Start reading must exist  
✅ Energy must never be negative  
✅ Zero means machine stopped  
✅ Voltage must be in valid range (200-250V)  
✅ Data gap > 10 seconds → Flag sensor issue  

### Validation Query:

```sql
-- Check for negative energy (indicates error)
SELECT 
  block_name,
  timestamp,
  total_active_energy
FROM energy_readings
WHERE total_active_energy < LAG(total_active_energy) OVER (PARTITION BY block_name ORDER BY timestamp)
ORDER BY timestamp DESC;
```

## 20. Business Use Cases

* ✅ Energy Cost per Unit Production
* ✅ Machine Performance Tracking
* ✅ Power Loss Detection
* ✅ Load Balancing
* ✅ Production vs Power Optimization

### Cost Calculation Example:

```sql
-- Calculate energy cost per machine per day
SELECT 
  block_name,
  DATE(timestamp) AS date,
  (MAX(total_active_energy) - MIN(total_active_energy)) AS energy_kwh,
  (MAX(total_active_energy) - MIN(total_active_energy)) * 0.35 AS cost_omr
FROM energy_readings
WHERE timestamp BETWEEN '2025-12-10 00:00:00' AND '2025-12-10 23:59:59'
GROUP BY block_name, DATE(timestamp);
```

### Production Efficiency:

```sql
-- Energy per ton of production
SELECT 
  er.block_name,
  (MAX(er.total_active_energy) - MIN(er.total_active_energy)) AS energy_consumed,
  SUM(pr.produced_weight) AS total_production,
  ((MAX(er.total_active_energy) - MIN(er.total_active_energy)) / SUM(pr.produced_weight)) AS energy_per_ton
FROM energy_readings er
LEFT JOIN production_records pr ON DATE(er.timestamp) = DATE(pr.created_at)
WHERE er.timestamp BETWEEN '2025-12-01' AND '2025-12-31'
GROUP BY er.block_name;
```

## 24. Database Indexes (CRITICAL for Performance)

```sql
-- Primary index for fast time-based queries
CREATE INDEX idx_energy_block_timestamp 
ON energy_readings (block_name, timestamp DESC);

-- Index for date queries
CREATE INDEX idx_energy_date 
ON energy_readings (DATE(timestamp), block_name);

-- Index for hourly aggregations
CREATE INDEX idx_energy_hour 
ON energy_readings (date_trunc('hour', timestamp), block_name);

-- Partial index for recent data (last 7 days)
CREATE INDEX idx_energy_recent 
ON energy_readings (block_name, timestamp DESC)
WHERE timestamp > NOW() - INTERVAL '7 days';
```

## 26. Alerting System

### Alert Conditions:

1. **High Energy Consumption**
   - Threshold: > 150% of average
   - Action: Send notification

2. **Machine Downtime**
   - Threshold: No data for > 10 minutes
   - Action: Alert maintenance team

3. **Voltage Fluctuation**
   - Threshold: < 200V or > 250V
   - Action: Urgent alert

4. **Negative Energy Delta**
   - Threshold: End < Start
   - Action: Flag data error

### Alert Query:

```sql
-- Find machines with unusually high consumption today
SELECT 
  block_name,
  (MAX(total_active_energy) - MIN(total_active_energy)) AS today_energy,
  AVG(des.energy_used) AS avg_7day_energy,
  ((MAX(total_active_energy) - MIN(total_active_energy)) - AVG(des.energy_used)) / AVG(des.energy_used) * 100 AS deviation_percent
FROM energy_readings er
LEFT JOIN daily_energy_summary des ON des.block_name = er.block_name
  AND des.date BETWEEN CURRENT_DATE - 7 AND CURRENT_DATE - 1
WHERE er.timestamp >= CURRENT_DATE
GROUP BY er.block_name
HAVING ((MAX(total_active_energy) - MIN(total_active_energy)) - AVG(des.energy_used)) / AVG(des.energy_used) > 0.5;
```

## 27. Integration with Production System

### Link Energy to Production:

```sql
-- Energy per ton of product
SELECT 
  pr.product_type,
  pr.shift,
  SUM(pr.produced_weight) AS total_production,
  (MAX(er.total_active_energy) - MIN(er.total_active_energy)) AS energy_consumed,
  ((MAX(er.total_active_energy) - MIN(er.total_active_energy)) / SUM(pr.produced_weight)) AS kwh_per_ton
FROM production_records pr
JOIN energy_readings er ON er.block_name = pr.machine_name
  AND DATE(er.timestamp) = DATE(pr.created_at)
WHERE pr.created_at BETWEEN '2025-12-01' AND '2025-12-31'
GROUP BY pr.product_type, pr.shift;
```
